# FAIRFluids Data Model Specification (Paper Draft)

## 1. Scope and Design Goals

The FAIRFluids data model is designed to represent fluid-property datasets in a way that is both human interpretable and machine actionable. Its central objective is to keep chemical identity, bibliographic provenance, measurement conditions, units, and sample history in one unified representation so that data can be reused across experiments, simulations, and digital repositories. The model supports both pure-component systems and multicomponent mixtures and is explicitly structured so that every reported value can be linked to the method and parameter context under which it was generated.

## 2. Top-Level Document Structure

A complete dataset is represented by a single FAIRFluidsDocument object. This root object contains a version section of type Version, a citation section of type Citation, a compound array of type Compound[], and a fluid array of type Fluid[]. The version element stores the schema/report version in versionMajor and versionMinor as integer fields. The citation element captures publication-level provenance, whereas the compound and fluid collections capture chemistry and measured system behavior, respectively. This separation allows one document to represent multiple fluid systems that originate from one publication while maintaining unambiguous global metadata.

## 3. Core Object Model

The Citation object stores source metadata and includes litType (LitType), author (Author[]), doi, page, pub_name, title, lit_volume_num, url_citation, and publication_year. In semantic terms, litType classifies the publication form, doi provides a persistent digital identifier, page stores the page interval of the cited source, pub_name records the journal or book title, and lit_volume_num stores volume information where available. url_citation gives a direct citation link and publication_year records the release year. Author identity is represented by the Author object through given_name and family_name, with optional but strongly useful identifiers such as email, orcid, and affiliation to improve attribution and disambiguation.

Chemical entities are represented by Compound objects. Each compound entry includes compoundID (Identifier) as an internal dataset key and pubChemID (integer) for external mapping to PubChem. Human-readable and machine-readable identities are jointly represented by commonName, name_IUPAC, SELFIE, standard_InChI, standard_InChI_key, and smiles_code. In addition, molar_weigth (float) stores the molar mass in g/mol, and sigma_profile (integer) stores sigma profile information used in downstream physicochemical modeling workflows. This design ensures that compounds remain traceable even when only partial naming information is available in source literature.

A Fluid object represents one fluid system and contains fluidID (Identifier[]), compounds (Identifier[]), property (Property[]), parameter (Parameter[]), and sample (Sample). In this context, compounds lists the participating chemical components by reference to Compound.compoundID; a single entry indicates a pure fluid, whereas multiple entries indicate a mixture. Reported target quantities are defined in the Property object with propertyID, properties, and unit. Here, propertyID is the local identifier, properties is the controlled vocabulary entry (for example density or viscosity), and unit is the explicit unit definition. Experimental or contextual inputs are defined in the Parameter object with parameterID, parameters, unit, and optional associated_compounds[]. This optional association is essential in mixture systems where a condition such as mole fraction is meaningful only for a specific component.

Sample-level context is captured by the Sample object, which includes sample_id (Identifier), associated_compounds[] (string), measurement (Measurement[]), storage (Storage), preparation (Preparation), and vendor_chemical (Vendor_Chemical). The field measurement is the core numerical payload and contains one or more Measurement entries. Each Measurement includes measurement_id (Identifier) as a unique data-point identifier, source_doi (string) for source-level provenance, propertyValue (PropertyValue[]) for dependent values, parameterValue (ParameterValue[]) for independent values, method (Method) to classify data generation mode, and method_description (string) as free-text detail of instrumentation, model, protocol, or literature extraction.

Each PropertyValue entry stores one dependent numeric value with properties (Properties), propertyID (Identifier), propValue (float), and uncertainty (float). The field propValue is the measured or calculated property magnitude, while uncertainty stores the associated error estimate, for example a standard uncertainty or confidence-derived spread. Each ParameterValue entry stores one conditioning value with parameters (Parameters), parameterID (Identifier), paramValue (float), and uncertainty (float). The paired representation of PropertyValue[] and ParameterValue[] within the same measurement enables one experimental state to carry multiple outputs and one output to be interpreted in a fully specified condition space.

Unit representation is explicit and decomposable. UnitDefinition includes unitID (string), name (string), and base_units (BaseUnit[]). The unitID is used for referencing, whereas name gives the readable unit label. Each BaseUnit entry contains kind (string) for the physical dimension, exponent (integer) for dimensional power, multiplier (float) for scalar conversion factors, and scale (float) for power-of-ten shifts. This allows deterministic reconstruction of complex derived units and reliable unit harmonization across heterogeneous source datasets.

Sample handling metadata is represented by Storage, StorageConditions, Vessel, Preparation, and Vendor_Chemical. Storage includes storage_type (StorageType), storage_conditions (StorageConditions), vessel (Vessel), time_prepared (string), and time_used (string). In StorageConditions, Temperature and Pressure define physical storage conditions, gassed indicates whether degassing was performed, inert indicates handling under inert atmosphere, and light indicates light exposure. Vessel includes id, name, volume, unit, and constant; the boolean constant defaults to true and specifies whether vessel volume is assumed constant. Preparation captures preparation protocol in prepMethod. Vendor_Chemical stores procurement metadata through assciciated_compound, CAS, purity, Vendor, and LOT, thereby preserving information that is often critical for reproducibility and batch-level comparison.

## 4. Datapoint Semantics and Linking Logic

In FAIRFluids, a datapoint is operationally represented by one Measurement instance. Its semantic content is defined jointly by reported outcomes (propertyValue[]), conditioning variables (parameterValue[]), and provenance metadata (method, method_description, and optionally source_doi). The method field distinguishes whether values are measured, calculated, simulated, or derived from literature. Because properties and parameters are represented as arrays, the model naturally supports multi-output measurements at shared conditions, for example density and viscosity reported for one temperature-pressure-composition state.

Referential consistency is achieved through identifier-based linking across levels. Fluid.compounds[] points to Compound.compoundID values, PropertyValue.propertyID points to Property.propertyID, and ParameterValue.parameterID points to Parameter.parameterID. Additional compound-specific assignment in mixtures is represented through associated_compounds[] in both Parameter and Sample. This mechanism is particularly important when composition-related descriptors such as mole fraction, molality, or solvent-ratio parameters are used.

## 5. Enumerations

The model uses controlled vocabularies to reduce ambiguity and improve interoperability. LitType contains book, journal, report, patent, thesis, conferenceProceedings, archivedDocument, personalCorrespondence, publishedTranslation, and unspecified. Method contains measured, calculated, simulated, and literature.

Properties includes thermophysical and related targets such as density, specificHeatCapacity, molarHeatCapacity, electricalConductivity, thermalConductivity, meltingPoint, boilingPoint, vaporPressure, compressibility, viscosity, kinematicViscosity, pH, polarity, surfaceTension, speedOfSound, refractiveIndex, diffusionCoefficient, molarVolume, specificVolume, excessMolarVolume, isobaricExpansionCoefficient, isothermalCompressibility, molarEnthalpy, molarEntropy, gibbsFreeEnergy, helmholtzFreeEnergy, excessMolarEnthalpy, excessMolarEntropy, excessMolarGibbsFreeEnergy, activationEnergy, henrysLawConstant, activityCoefficient, fugacityCoefficient, osmoticCoefficient, activity, criticalTemperature, criticalPressure, criticalDensity, criticalVolume, triplePointTemperature, triplePointPressure, criticalPointTemperature, criticalPointPressure, and ionicStrength. These entries cover transport, volumetric, thermodynamic, equilibrium, critical, transition, and selected solution-chemistry observables.

Parameters includes condition and composition descriptors including Time, Temperature, Upper temperature, Lower temperature, Pressure, Partial pressure, Upper pressure, Lower pressure, Mole fraction, Mass fraction, Molality, Amount concentration (molarity), Volume fraction, Ratio of amount of solute to mass of solution, Ratio of mass of solute to volume of solution, Amount ratio of solute to solvent, Mass ratio of solute to solvent, Volume ratio of solute to solvent, Initial mole fraction of solute, Final mole fraction of solute, Initial mass fraction of solute, Final mass fraction of solute, Initial molality of solute, Final molality of solute, Solvent: Mole fraction, Solvent: Mass fraction, Solvent: Volume fraction, Solvent: Molality, Solvent: Amount concentration (molarity), Solvent: Amount ratio of component to other component of binary solvent, Solvent: Mass ratio of component to other component of binary solvent, Solvent: Volume ratio of component to other component of binary solvent, Solvent: Ratio of amount of component to mass of solvent, Solvent: Ratio of component mass to volume of solvent, Wavelength, Frequency, Molar volume, Specific volume, Mass density, Amount density, Molar entropy, Relative activity, Activity coefficient, Amount, and Mass. StorageType includes Fresh, Fridge, Open, Closed, and Dessicator, allowing standardized storage-state annotation.

## 6. Minimal Completeness for Publication Use

For publication-quality exchange, a record should at minimum include document-level metadata (version, citation, at least one compound, and at least one fluid), traceable chemical identity (compoundID, a compound name, and at least one structural identifier such as InChI or SMILES), explicit property and parameter definitions (propertyID, properties, unit, parameterID, parameters, unit), and at least one fully specified measurement (measurement_id, one or more propertyValue entries, one or more parameterValue entries, and method). Reproducibility is substantially improved when source identifiers (doi or source_doi) are present and method_description contains enough technical detail to reconstruct how values were obtained.

## 7. Interoperability Considerations

The FAIRFluids model intentionally combines normalized references (identifier fields) with controlled semantic labels (properties, parameters) to support both strict data integration and user-facing readability. Its explicit unit decomposition enables deterministic unit conversion and cross-dataset harmonization. For mixture data, consistent population of compound-association fields is essential to preserve the physical meaning of composition-dependent variables.
