# ThermoML-to-FAIRFluids Mapping

This document describes, in paper-style prose, the mapping strategy implemented in the `thermoml_to_fairfluids` package for transforming ThermoML XML records into FAIRFluids-compliant JSON documents. The converter is designed as a deterministic, three-layer pipeline that separates syntactic extraction, semantic normalization, and target-schema construction. This layered organization makes the transformation reproducible and extensible while preserving the provenance and measurement semantics contained in the source files.

The entry point of the workflow is the `convert()` function in `converter.py`, which executes three consecutive steps: parsing the source XML into raw data structures, normalizing those structures into a canonical intermediate representation, and constructing FAIRFluids objects using controlled mappings for properties, parameters, units, and methods. This sequence can be summarized as ThermoML XML -> RawThermoML -> CanonicalDataset list -> FAIRFluids JSON.

In the first layer (`parser.py`), the converter performs namespace-aware extraction of ThermoML elements (`http://www.iupac.org/namespaces/ThermoML`) and stores them in `Raw*` Pydantic models defined in `canonical_model.py`. At this stage, no semantic interpretation is applied. The parser reads citation metadata (`Citation`), compound records (`Compound`), and each experimental block (`PureOrMixtureData`) including components, properties, variables, constraints, and numerical values. Importantly, property names and variable/constraint type strings are kept as-is, because these strings are later resolved by dedicated mapping modules. Numerical values are extracted together with available uncertainties. For properties, uncertainties are read preferentially from combined uncertainty fields (`nCombExpandUncertValue`, then `nCombStdUncertValue`) and otherwise from standard uncertainty fields (`nStdUncertValue`). Variable uncertainties are extracted from `VarUncertainty` when present. The output of this layer is therefore a faithful structural projection of the ThermoML file.

The second layer (`canonical_builder.py`) converts raw records into a normalized internal representation. This step introduces stable internal identifiers and harmonizes ThermoML constructs that are represented separately in the source schema. Specifically, all measured independent variables (`Variable`) and fixed experimental conditions (`Constraint`) are unified into a single parameter space (`CanonicalParameter`). Variable-derived parameters are identified as `param_<nVarNumber>`, while constraints are represented as `constraint_<nConstraintNumber>` with `is_constraint=True` and an embedded fixed value. This design allows downstream code to treat all independent conditions uniformly during measurement construction. Properties are similarly normalized as `CanonicalProperty` objects with IDs `prop_<nPropNumber>`.

A key operation in this layer is row synthesis from `NumValues` blocks. Each row (`CanonicalRow`) stores parameter values, property values, and uncertainties in dictionary form keyed by canonical IDs. Constraint values are injected into every row, which ensures that fixed conditions are explicitly represented for each measurement record in the target format. The canonical layer also creates compound references using safe name-based IDs (for example, converting names to lowercase and replacing whitespace/punctuation), enabling dataset-local linking before FAIRFluids IDs are generated.

The third layer (`fairfluids_builder.py`) materializes FAIRFluids objects from canonical datasets and raw metadata. A fresh `IDRegistry` instance (`id_registry.py`) generates deterministic, monotonically increasing IDs per object category (`compound_1`, `property_1`, `parameter_1`, `measurement_1`, etc.) during a single conversion run. The final document follows the FAIRFluids root structure with `citation`, `compound`, and `fluid` sections. Citation fields are mapped directly where possible (title, journal, year, volume, page), with DOI normalization to remove URL prefixes (`doi.org/...`) and conversion to a canonical DOI string. Author strings are parsed into structured author objects (`given_name`, `family_name`) by splitting entries on semicolons and comma-separated name parts.

Each canonical dataset becomes one FAIRFluids `Fluid` object. Within a fluid, properties and parameters are defined once and then referenced by per-row measurement values. This strategy preserves the distinction between schema-level definitions (what was measured, under which parameter types and units) and instance-level observations (the numerical values for each row). The sample object stores the measurement list and associated compound references. Source DOI and inferred method information are propagated into each measurement, so that each numerical record remains traceable to literature provenance.

Semantic mapping is implemented through three dedicated modules in `mappers/`: `property_mapper.py`, `parameter_mapper.py`, and `unit_mapper.py`. The property mapper resolves ThermoML `ePropName` strings to FAIRFluids `Properties` enum members. It first attempts exact dictionary lookup, then normalized string matching, and finally direct comparison against enum values. The mapping table covers transport, thermodynamic, phase-equilibrium, critical-point, and composition-related properties, including common lexical variants (for example, alternate typography in units or naming variants such as “Gibbs free energy” versus “Gibbs energy”). If no match is found, the mapper returns `None` and emits a warning, allowing conversion to continue while signaling unmapped semantics.

The parameter mapper resolves ThermoML variable and constraint type strings to FAIRFluids `Parameters` enum members. It supports primary state variables (temperature, pressure), composition descriptors (mole/mass/volume fraction), concentration and ratio terms, solvent-specific composition descriptors, and selected spectroscopic descriptors (frequency, wavelength). Lookup is performed by exact mapping table match, with fallback to case-insensitive enum value matching.

Unit handling is performed by the unit mapper, which constructs FAIRFluids `UnitDefinition` objects from either complete ThermoML property strings or explicit unit substrings. The mapper includes a curated unit dictionary with decomposed SI base-unit representations (`BaseUnit`) and supports alias normalization (for example, `Pa s` -> `Pa*s`, `m^2/s` -> `m2/s`, `kg/m^3` -> `kg/m3`). For composition fractions (mole, mass, and volume fraction), the builder enforces a dimensionless unit representation. If a unit cannot be resolved, the mapper still returns a valid `UnitDefinition` with the raw unit label and an empty `base_units` list, preserving information while making uncertainty in unit semantics explicit.

Method mapping is performed in `fairfluids_builder.py` through a controlled vocabulary that translates textual method hints (e.g., “measured,” “molecular dynamics,” “ab initio,” “correlation”) to FAIRFluids `Method` enum values (`MEASURED`, `SIMULATED`, `CALCULATED`, `LITERATURE`). If method information is absent or ambiguous, the converter defaults to `LITERATURE`. This fallback reflects the primary use case of literature-derived ThermoML records while maintaining a consistent output schema.

Conceptually, the transformation bridges two different data models. ThermoML encodes many semantics in nested XML structures and text-bearing fields, whereas FAIRFluids expects explicit object definitions, enum-constrained descriptors, and typed unit definitions. The canonical intermediate layer is therefore essential: it decouples XML traversal from semantic mapping, minimizes cross-layer coupling, and enables robust handling of mixed variable/constraint experimental designs. By injecting fixed constraints into each row and preserving per-value uncertainties, the converter produces FAIRFluids outputs that are both machine-actionable and faithful to original experimental context.

The command-line interface in `main.py` exposes this workflow as:

`python -m thermoml_to_fairfluids.main <thermoml.xml> [output.json]`

When an output path is provided, the result is serialized as formatted JSON; otherwise, it is printed to standard output. Programmatic usage is available via `from thermoml_to_fairfluids import convert`.

In summary, the implemented mapping strategy combines deterministic ID generation, explicit semantic crosswalks, and conservative fallback behavior to convert heterogeneous ThermoML records into FAIRFluids-compliant structured data. The approach is designed to be transparent for methodological reporting, practical for batch conversion, and extensible for future additions to property/parameter vocabularies and unit coverage.
